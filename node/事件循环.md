下面用一个“收到请求后读文件、安排日志和超时”的小案例，把 Node 事件循环走一遍。官方说法里，事件循环让 Node 在默认单个 JS 线程上做非阻塞 I/O：I/O 尽量交给系统内核或 libuv，完成后再把回调放回事件循环队列里执行。参考：[Node.js Event Loop](https://nodejs.org/learn/asynchronous-work/event-loop-timers-and-nexttick#what-is-the-event-loop) ([nodejs.org](https://nodejs.org/learn/asynchronous-work/event-loop-timers-and-nexttick))

```js
// event-loop-demo.cjs
const fs = require("node:fs");

console.log("A: script start");

fs.readFile(__filename, () => {
  console.log("D: fs.readFile callback - poll 阶段的 I/O 回调");

  setTimeout(() => {
    console.log("H: setTimeout(0) - timers 阶段");
  }, 0);

  setImmediate(() => {
    console.log("G: setImmediate - check 阶段");
  });

  process.nextTick(() => {
    console.log("E: process.nextTick - 当前回调结束后立刻执行");
  });

  Promise.resolve().then(() => {
    console.log("F: Promise.then - microtask");
  });

  console.log("D2: fs callback sync end");
});

setTimeout(() => {
  console.log("C: top-level setTimeout(0)");
}, 0);

console.log("B: script end");
```

常见输出大致是：

```text
A: script start
B: script end
C: top-level setTimeout(0)
D: fs.readFile callback - poll 阶段的 I/O 回调
D2: fs callback sync end
E: process.nextTick - 当前回调结束后立刻执行
F: Promise.then - microtask
G: setImmediate - check 阶段
H: setTimeout(0) - timers 阶段
```

关键过程是：

1. 先执行同步代码  
   `A` 和 `B` 立刻输出。`fs.readFile` 不会马上读完，它把文件读取交出去，然后 JS 主线程继续跑。

2. 第一轮事件循环处理 timer  
   顶层 `setTimeout(..., 0)` 到期后进入 `timers` 阶段，所以输出 `C`。注意 `0ms` 不是“马上执行”，而是“最早过 0ms 后，在合适阶段执行”。官方也强调 timer 是阈值，不是精确时间。([nodejs.org](https://nodejs.org/learn/asynchronous-work/event-loop-timers-and-nexttick))

3. 文件读完后，I/O 回调进入 `poll` 阶段  
   `fs.readFile` 完成后，它的回调在 `poll` 阶段执行，所以输出 `D`。在这个回调里又安排了四类任务：`setTimeout`、`setImmediate`、`process.nextTick`、`Promise.then`。

4. 当前 I/O 回调必须先同步跑完  
   所以 `D2` 先输出。事件循环不会在一个回调执行到一半时切出去。

5. 回调结束后，先清空 `nextTick`，再清空 Promise microtask  
   所以 `E` 在 `F` 前面。`process.nextTick()` 不属于事件循环某个阶段，它是在当前 JS 调用栈清空后、事件循环继续前执行；Node 文档也说明 nextTick 队列会在 microtask 队列前处理。参考：[Node process.nextTick docs](https://nodejs.org/api/process.html#processnexttickcallback-args) ([nodejs.org](https://nodejs.org/api/process.html?utm_source=openai))

6. 因为 `setImmediate` 是在 I/O 回调里安排的，所以它通常先于这里的 `setTimeout(0)`  
   I/O 回调属于 `poll` 阶段，`setImmediate` 进入紧随其后的 `check` 阶段，输出 `G`。而 `setTimeout(0)` 要等 `timers` 阶段，输出 `H`。Node 官方也明确说：如果二者在 I/O 回调里调度，`setImmediate()` 会先执行。([nodejs.org](https://nodejs.org/learn/asynchronous-work/event-loop-timers-and-nexttick))

可以把 Node 事件循环想成这条主线：

```text
同步代码
  ↓
nextTick 队列
  ↓
Promise / queueMicrotask 队列
  ↓
事件循环阶段：
timers → pending callbacks → poll → check → close callbacks
```

实际开发里最重要的结论是：**异步 I/O 不阻塞 JS 主线程，但你的 JS 回调本身会阻塞事件循环。**  
如果你在 `fs.readFile`、HTTP 请求回调、Promise 回调里写了很重的 CPU 计算，后面的 `setTimeout`、新请求、socket 数据、`setImmediate` 都会被拖住。这也是为什么 Node 适合高并发 I/O，但 CPU 密集任务通常要拆分、丢给 Worker Threads，或者放到独立服务里。
